# 💰 v2.0.0 - Monetização

**Status:** 💡 **IDEIA** (aguardando validação de mercado)
**Estimativa:** 6-8 semanas
**Prioridade:** 🔵 BAIXA (após v1.1 e v1.2)

---

## 🎯 Objetivo da Release

Implementar sistema de assinatura recorrente para sustentabilidade do projeto, com tiers de features e gateway de pagamento integrado.

---

## 💳 Modelo de Negócio

### Planos Propostos:

| Plano | Preço | Features |
|-------|-------|----------|
| **Free** | R$ 0 | Resumos semanais, dashboard básico, até 3 provas cadastradas |
| **Pro** | R$ 9,90/mês | Insights IA, análises avançadas, provas ilimitadas, upload de comprovantes, sem anúncios |
| **Teams** | R$ 29,90/mês | Tudo do Pro + grupos de corrida, ranking entre amigos, desafios semanais, coaching via WhatsApp |

### Estratégia de Pricing:
- **Freemium:** Core features gratuitas para atrair usuários
- **Pro:** Para corredores sérios que querem evoluir
- **Teams:** Para grupos de corrida / assessorias esportivas

### 💳 1. Sistema de Assinatura

#### 📋 Checklist:

**Banco de Dados:**
- [ ] Criar tabela `choppinho.subscriptions`:
  ```sql
  CREATE TABLE choppinho.subscriptions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES choppinho.users(id) ON DELETE CASCADE,
    plan_type VARCHAR(20) NOT NULL, -- 'free', 'pro', 'teams'
    status VARCHAR(20) NOT NULL, -- 'active', 'cancelled', 'past_due', 'trial'

    -- Gateway (Stripe ou Abacatepay)
    gateway VARCHAR(20) NOT NULL, -- 'stripe', 'abacatepay'
    gateway_customer_id VARCHAR(255),
    gateway_subscription_id VARCHAR(255),

    -- Datas
    trial_ends_at TIMESTAMPTZ,
    current_period_start TIMESTAMPTZ NOT NULL,
    current_period_end TIMESTAMPTZ NOT NULL,
    cancelled_at TIMESTAMPTZ,

    -- Preço
    price_brl DECIMAL(10,2) NOT NULL,

    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
  );

  CREATE INDEX idx_subscriptions_user ON choppinho.subscriptions(user_id);
  CREATE INDEX idx_subscriptions_gateway ON choppinho.subscriptions(gateway_subscription_id);
  ```

- [ ] Criar tabela `choppinho.payments`:
  ```sql
  CREATE TABLE choppinho.payments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    subscription_id UUID NOT NULL REFERENCES choppinho.subscriptions(id),
    user_id UUID NOT NULL REFERENCES choppinho.users(id),

    amount_brl DECIMAL(10,2) NOT NULL,
    status VARCHAR(20) NOT NULL, -- 'pending', 'paid', 'failed', 'refunded'

    gateway VARCHAR(20) NOT NULL,
    gateway_payment_id VARCHAR(255),
    gateway_charge_id VARCHAR(255),

    payment_method VARCHAR(50), -- 'credit_card', 'pix', 'boleto'

    paid_at TIMESTAMPTZ,
    refunded_at TIMESTAMPTZ,

    metadata JSONB,

    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
  );

  CREATE INDEX idx_payments_subscription ON choppinho.payments(subscription_id);
  CREATE INDEX idx_payments_user ON choppinho.payments(user_id);
  ```

### 💰 2. Gateway de Pagamento

#### Opção A: **Stripe** (Internacional + Cartão)

**Vantagens:**
- API robusta e bem documentada
- Suporte global
- Webhooks confiáveis
- Portal de gerenciamento pronto

**Desvantagens:**
- Taxa: 4.99% + R$ 0.39 por transação
- Foco em cartão (PIX via parceiros)

**Checklist:**
- [ ] Criar conta Stripe Brasil
- [ ] Configurar produtos e preços no dashboard
- [ ] Implementar Stripe Checkout
- [ ] Configurar webhooks (assinatura, pagamento, cancelamento)
- [ ] Testar com cartões de teste

---

#### Opção B: **Abacatepay** (PIX Nativo BR)

**Vantagens:**
- PIX nativo (instantâneo)
- Taxa: 2.99% (menor que Stripe)
- Foco no mercado brasileiro
- Checkout simples

**Desvantagens:**
- API menos madura que Stripe
- Documentação limitada
- Menos features (sem portal de gerenciamento)

**Checklist:**
- [ ] Criar conta Abacatepay
- [ ] Configurar planos via API
- [ ] Implementar geração de PIX
- [ ] Webhook de confirmação de pagamento
- [ ] Testar com PIX de teste

---

#### Estratégia Híbrida (Recomendado):
- **Stripe:** Para quem prefere cartão
- **Abacatepay:** Para quem prefere PIX
- UI permite escolher método de pagamento

## 📈 Métricas de Sucesso

### KPIs de Monetização:
- **Taxa de conversão Free → Pro:** 5% (mês 1) → 10% (mês 6)
- **Churn rate:** < 5% ao mês
- **LTV (Lifetime Value):** R$ 120+ (12 meses Pro)
- **MRR (Monthly Recurring Revenue):** R$ 10.000 (1000 Pro users)

### KPIs de Produto:
- **Trial → Paid:** 30% dos trials viram assinantes pagos
- **Grupos criados (Teams):** 10+ grupos ativos
- **Desafios completados:** 50+ por semana

## 🐛 Riscos e Mitigações

| Risco | Impacto | Mitigação |
|-------|---------|-----------|
| Gateway de pagamento instável | Crítico | Fallback entre Stripe/Abacatepay, retry logic |
| Webhooks perdidos/duplicados | Alto | Idempotência, logging, retry manual |
| Fraude/chargebacks | Médio | Validação de email, análise de padrão |
| Churn alto (cancelamentos) | Alto | Pesquisa de feedback, melhorar onboarding |

## 🚀 Deployment

### Pré-requisitos:
- [ ] Conta Stripe configurada (modo produção)
- [ ] Conta Abacatepay configurada
- [ ] Variáveis de ambiente:
  ```
  STRIPE_SECRET_KEY=sk_live_xxx
  STRIPE_WEBHOOK_SECRET=whsec_xxx
  ABACATEPAY_API_KEY=xxx
  ABACATEPAY_WEBHOOK_SECRET=xxx
  ```

### Ordem de Deploy:
1. **Migration SQL** (criar tabelas de subscription/payments)
2. **Backend** (endpoints + webhooks)
3. **Frontend** (tela de plano)
4. **Testar em staging** (modo sandbox dos gateways)
5. **Deploy produção** (modo live)
6. **Monitorar webhooks** (primeiras 48h críticas)

### Rollback Plan:
- Manter plano Free funcional mesmo se monetização falhar
- Migração reversa: `DROP TABLE subscriptions CASCADE`
